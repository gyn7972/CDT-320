CDT-310 → CDT-320 Vision 이식 가이드

실제 구동 장비(CDT-310)의 비전 구현 방식과 신규 장비(CDT-320) 적용 방법을 코드 분석 기반으로 비교
대상: QMC.Vision 레시피 재구성 · 작성 기준 실제 소스 코드(C:\Project\CDT-310, C:\Project\CDT-320) · 2026-06-18
CDT-310 (구현 방식) CDT-320 (적용 방법) 이식 가능 그대로/유사 재사용 구현 필요 로직 별도 이식 제거/핸들러 Vision 비포함
한눈 요약 — CDT-310은 상속 기반(VisionPart → PatternMatchingVisionPart → 각 기능)으로 비전 기능마다 클래스를 두고 Cognex VisionPro를 직접 래핑했습니다. CDT-320은 인터페이스 + 노드 기반(IPatternFinder / IInspector + VisionModule의 AddFinder/AddInspector)으로 바뀌어, 모든 알고리즘이 모듈의 자식 노드가 되고 데이터는 레시피(JSON) 단위로 cascade 저장됩니다. 핵심 이식 원칙은 “Vision은 측정·계산만, 모션·구동은 핸들러” 그리고 “모든 레시피 데이터는 활성 Recipe를 따라간다” 입니다.

1. 아키텍처 철학 비교

  • 백엔드 = Cognex VisionPro를 VisionProPatternMatchingVisionTool로 직접 래핑

  • 검사(Inspector)는 별도 프로젝트 (CDTInspector, SideChippingInspector) — byte[,] 영상 기반, CUDA

CDT-320

인터페이스 + 노드 기반

└ AddFinder() → FinderAlgorithm 노드
└ AddInspector() → InspectorAlgorithm 노드
  • 알고리즘 = IPatternFinder / IInspector 인터페이스 구현체
  • 백엔드 교체형: Cognex → OpenCV → Sim 자동 fallback (IVisionBackend)
  • 데이터 = FinderAlgo{Setup,Config,Recipe} POCO, DataContract JSON
  • 저장 = 모듈 1회 호출로 자식 노드 전체 cascade (Recipes/{명칭}/{키}.recipe.json)
QMC.Vision/Equipment/Core, /Unit, /Backends
이식의 큰 그림: 310의 OnTrain/OnSearch 패턴 매칭 “알맹이”는 320 IPatternFinder.Train()/Match()에 그대로 들어갑니다. 바뀌는 것은 껍데기(상속 → 노드 등록)와 저장 방식(바이너리 → JSON 레시피), 그리고 모션 분리입니다.

CDT-320은 5개 모듈로 고정되고, 각 모듈이 AddFinder/AddInspector로 노드를 등록합니다. 310의 어떤 기능이 어디로 들어가는지:

4. 데이터 / 레시피 저장 비교

CDT-310
  • 각 Part의 Recipe/Config 클래스 ([Serializable])
  • 바이너리 직렬화 — Owner.SaveRecipeData()
  • 파라미터: PatternMatchingParameters(MinScore, AngleTolerance, MaxInstance…)
  • 스케일/오프셋은 Config(VisionScale, GridResults)에 저장
CDT-320
  • POCO FinderAlgo{Setup,Config,Recipe} + DataContract JSON
  • 경로 Recipes/{recipeName}/{StorageKey}.recipe.json (예 WaferVision.EjectPinFinder.recipe.json)
  • 모듈 1회 SaveRecipe(명칭) → 자식 노드 전체 cascade
  • ApplyToRuntime()/CollectFromRuntime()로 POCO ↔ 런타임 동기화
호환성 주의: 직렬화 포맷이 바이너리 → JSON으로 바뀌므로 310 레시피 파일은 그대로 못 읽습니다. 재티칭 또는 1회 마이그레이션이 필요합니다. 또한 에디터가 활성 레시피명을 따라가야 합니다(현재 "default" 고정 버그 — RECIPE_LOAD_POLICY 문서 참조).

5. 이식 시 핵심 주의점

θ/offset 계산 CUDA Inspector Compensator 제거 Edge/Histogram/Color 미연결 모션 분리 직렬화 포맷
  • 2점 정렬 θ 계산 누락 금지 — 320 Finder는 점 위치만 찾음. 310 Aligner.TwoPointAlign의 각도/오프셋 계산을 핸들러 또는 모듈 단계로 반드시 이식.
  • Inspector 로직 깊이 차이 — 320 CognexInspector는 단순 Blob 면적 PASS 기준. 310 CDTInspector/SideChippingInspector의 chipping·foreign·임계 로직을 옮겨야 실제 검사가 됨.
  • Edge/Histogram/Color 백엔드 미연결 — 인터페이스만 정의됨. IVisionBackend는 CreatePatternFinder/CreateInspector 둘만 생성. 측면 칩핑을 EdgeFinder로 하려면 백엔드 팩토리부터 추가 필요.
  • 모션은 전부 핸들러 — Jog/Stage move/Compensator 제거. AutoFocus(=Z), AutoScale(=이동)은 “구동=핸들러, 측정=Vision” 콜백으로.
  • 레시피 SSOT — 모든 레시피 화면 값은 활성 Recipe를 따라 read/write. 수동 편집 + 핸들러 수신값 양방향.

6. CDT-320에서 신규 Finder/Inspector 추가 절차

310 기능을 320에 넣을 때의 표준 작업 순서 (코드 확인 기반):

1
데이터 타입 결정 — 공용 FinderAlgo{Setup,Config,Recipe} 재사용. 전용 파라미터가 필요하면 파생 Recipe 정의.
2
모듈에 등록 — 모듈 생성자에서 AddFinder<...>("XxxFinder") / AddInspector<...>("XxxInspector"). 자동으로 Finders/Inspectors + Components(cascade) 등록.
3
백엔드 구현 확인 — CognexPatternFinder/CognexInspector가 Train/Match/Inspect 제공. 전용 로직은 IAlgoParamSync로 POCO↔런타임 동기화.
4
UI 자동 반영 — RecipePage가 module.Finders/Inspectors를 열거 → 세팅선택기에 자동 노출. 라벨은 InspectionLabel에 추가.
5
파라미터 그리드 — VisionTargetPage/InspectorTargetPage의 AppendNodeParams()에 전용 필드만 추가(공용은 자동).
6
저장/로드 검증 — LoadRecipe(명칭)→ApplyToRuntime, SaveRecipe(명칭)→CollectFromRuntime cascade. tools/verify_vision_features.pl 회귀.
// 예: WaferVisionModule 생성자 — 노드 등록 한 줄이면 끝 (UI·저장 자동)
public WaferVisionModule(ICamera camera, IVisionBackend backend)
    : base("WaferVision", camera, backend)
{
    EjectPin        = AddFinder<FinderAlgoSetup, FinderAlgoConfig, FinderAlgoRecipe>("EjectPinFinder");
    Reticle         = AddFinder<...>("ReticleFinder");
    FirstReference  = AddFinder<...>("FirstReferenceFinder");
    // ... 각 호출이 Finders 딕셔너리 + Components(cascade 저장) 자동 등록
}
// 노드 ↔ 런타임 동기화 (AlgorithmNode) — 저장 포맷이 레시피 JSON으로 일원화
protected override void ApplyToRuntime() {   // LoadRecipe 직후: POCO → 런타임
    if (Recipe is FinderAlgoRecipe r) {
        _finder.SearchRoi = r.SearchRoi?.Clone();
        _finder.TrainRoi  = r.TrainRoi?.Clone();
        _finder.AcceptThreshold = r.AcceptThreshold;
    }
}
protected override void CollectFromRuntime() { // SaveRecipe 직전: 런타임 → POCO
    if (Recipe is FinderAlgoRecipe r) {
        r.SearchRoi = _finder.SearchRoi?.Clone();
        r.TrainRoi  = _finder.TrainRoi?.Clone();
        r.AcceptThreshold = _finder.AcceptThreshold;
    }
}
결론: 310의 알고리즘 로직(패턴 매칭, 칩핑 검사, 스케일 계산식)은 320의 Train/Match/Inspect 안으로 옮기고, 껍데기(상속→노드 등록)와 저장(바이너리→레시피 JSON)은 320 프레임워크가 처리합니다. 단 θ계산·정교한 Inspector·px→mm 변환은 명시적으로 이식해야 하고, 모션·Compensator는 Vision에서 제외합니다.
근거: CDT-310 (QMC.Common/VisionPart, CDT-310_1 Inspector), CDT-320 (QMC.Vision/Equipment/Core·Unit·Backends, Ui/Pages/Recipe). 코드에서 직접 확인된 사실 기반이며, 일부 로직 위치(θ계산·px→mm)는 구현 팀과 최종 확인 권장.
