QMC.Vision 시퀀서 구조

네임스페이스 QMC.Vision.Sequencing · 분석 기준: 장비PC(V:) 실제 소스 · 핸들러 시퀀서를 “유닛→모듈”로 미러

핵심 한 줄. Vision 시퀀서는 핸들러(TCP) 없이도 비전 PC가 스스로 검사 사이클을 돌리는 “Sim 자체실행” 경로입니다. 실제 운영(핸들러→TCP)과 똑같은 명령 코어(VisionCommandCore)를 공유해 결과·좌표변환·로그가 동일하고, 검사 NG 판정·MATCH score 게이트·chipUid 자재추적까지 실제 흐름과 같게 태웁니다. 모션은 포함하지 않습니다.

3. 모듈 1사이클 — 실행과 판정

RunModuleCycleAsync() 가 CycleSteps() 단계를 순서대로 실행하며, 각 단계 결과를 판정합니다.

  1. chipUid 발급 — SimEmitChipUid=true면 "SIM-{모듈}-{시각}" 합성 발급 → args[1]로 전달 → 실제 흐름처럼 MaterialTracker·이미지/데이터 로그까지 태움(아니면 빈 값=추적 생략).
  2. 단계 순회 — 모듈에 없는 finder/inspector는 조용히 skip.
  3. Context.Dispatch(Module, cmd, [id(, chipUid)]) → 코어 실행, 결과 문자열 수신.
  4. 실행 실패 판정 — 결과가 fail: 또는 ERR로 시작하면 NG. 사유를 사람이 읽기 쉬운 문구로 변환(예: 학습 없음→“학습 필요”).
  5. MATCH score 게이트 — MATCH가 OK;…score=여도 score < finder AcceptThreshold면 NG(score 표시). (Threshold ≤ 0이면 게이트 안 함)
  6. 사이클 NG 요약 — 실패 단계들을 한 줄로 모아 기록(원인 파악 용이).
  7. 스톱워치로 LastCycleMs·CycleCount 갱신(메트릭).

기본 CycleSteps() = GRAB → 모든 MATCH → 모든 INSPECT. 모듈별 override로 CDT-310 순서 정의.

4. 결과 판정 규칙 (중요)

“실행 실패”와 “검사 NG”는 다릅니다. 문자열 형식으로 구분하므로 혼동하면 안 됩니다.
결과 형식의미처리
fail:<사유> (소문자 콜론)실행 실패 — 학습 없음/모듈 미지정 등NG + 사람이 읽는 사유 + 알람
ERR…실행 오류(예외)NG + 로그
FAIL;… (대문자 세미콜론)검사 NG 판정 — 정상 결과(불량)실행 실패 아님(정상 처리)
OK;…score=0.93MATCH 성공score < AcceptThreshold면 NG

로그 정책: Auto는 실행 실패만 기록(연속실행 로그 폭주 방지), Manual/Step은 전 단계 기록. NG(검사/score)는 사유와 함께 별도 라인으로 남김.

5. 실행 모드 — Auto / Manual / Step

Auto

선택 모듈 연속 실행. 사이클마다 CycleIntervalMs 대기. 정상 로그 생략.

Manual / Step

게이트(SemaphoreSlim) 대기. StepUnit() 1회=1사이클. 직전 step 진행/대기 중이면 중복 무시.

병렬성

Coordinator가 선택 모듈마다 별도 Task로 동시 실행. 공유 토큰으로 일괄 정지(+CTS Dispose).
